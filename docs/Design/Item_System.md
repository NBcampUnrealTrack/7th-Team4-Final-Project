## 1. 아이템 시스템 (Item System)

게임 내 아이템은 크게 **장비 아이템**과 **소비 아이템**으로 분류된다.

| 아이템 종류 | 상세 타입 (Type) | 등급 (Grade) | 기본 성능 (Base Stat) | 랜덤 추가 옵션 (Bonus Options) | 주요 특징 및 효과 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **장비 아이템** | 무기 (Weapon) | 일반 (Common)<br>희귀 (Rare) | **공격력 (STR)** 증가 | **Common**: 없음<br>**Rare**: 1~2개 무작위 부여<br>*(공격력 / 방어력 / 최대 HP 증가)* | 캐릭터에 착용하여 능력치를 영구적으로 높인다. |
| **장비 아이템** | 갑옷 (Chest) | 일반 (Common)<br>희귀 (Rare) | **방어력 (DEF)** 증가 | **Common**: 없음<br>**Rare**: 1~2개 무작위 부여<br>*(공격력 / 방어력 / 최대 HP 증가)* | 캐릭터에 착용하여 능력치를 영구적으로 높인다. |
| **소비 아이템** | 물약 (Potion) | 등급 없음 | 없음 | 없음 | 사용 시 **5초 동안 매초 전체 체력(HP)의 3~5%씩** 서서히 회복. (이동 및 전투 중 사용 가능) |

## 2. 인벤토리 및 장비 슬롯 시스템

### 2.1 컴포넌트 구조

- **장비 컴포넌트 (`UEquipmentComponent`)**: 현재 장착 중인 장비 슬롯 관리
  - 보유 슬롯: 무기(Weapon) 1개, 갑옷(Chest) 1개
- **인벤토리 컴포넌트 (`UInventoryComponent`)**: 보유 아이템 가방 관리
  - 가방 크기: 총 30칸 (5 × 6 격자)

### 2.2 인벤토리 보관 규칙

- **소비 아이템 (물약)**: 동일 물약은 하나의 슬롯에 수량(`quantity`)만 증가 (스택)
- **장비 아이템 (무기/갑옷)**: 종류가 같더라도 무조건 개별 칸 점유 (옵션이 다를 수 있으므로)
- **아이템 소실**: 수량이 0이 되면 해당 인벤토리 슬롯 데이터 자동 삭제(빈칸 처리)

### 2.3 장착 및 해제 규칙

인벤토리 창에서 아이템을 **더블 클릭** 또는 **우클릭**하여 장착/해제를 실행한다.

- **장착 (Equip)**
  - 인벤토리 → 장비 컴포넌트 해당 슬롯으로 이동
  - 기본 성능 + 추가 옵션이 캐릭터 능력치에 즉시 반영
  - 슬롯에 이미 장비가 있으면 기존 장비는 인벤토리 빈 슬롯으로 자동 스왑
- **해제 (Unequip)**
  - 장비 능력치 즉시 차감
  - 벗은 장비는 인벤토리 빈 슬롯으로 이동
  - **인벤토리가 가득 차면 해제 불가**

## 3. 상호작용 시스템 (Interaction System)

### 3.1 조작법

- 대상 **1미터 이내 접근** 후 **F 키**로 실행
- 바닥에 떨어진 **아이템**은 마우스 **좌클릭**으로 즉시 획득

### 3.2 대상별 상호작용 효과

| 대상 | 효과 |
|------|------|
| 아이템 | 획득 → `UInventoryComponent`에 추가 |
| NPC | 대화 창 오픈 |
| 구조물 (보물상자, 세이브 포인트 등) | 오브젝트 고유 특수 이벤트 발생 |

### 3.3 상호작용 취소 규칙

| 조건 | 처리 |
|------|------|
| ESC 버튼 | 진행 중이던 대화/상호작용 즉시 종료 |
| 피격 (HP 감소) | 진행 중인 상호작용 강제 즉시 취소 |

## 4. 데이터 구조 명세

### FItemData 구조체

| 필드명 | 타입 | 설명 |
| :--- | :--- | :--- |
| `item_id` | FName | 아이템 고유 식별자 |
| `item_name` | FText | 표시 이름 |
| `item_category` | EItemCategory | 대분류 (Equipment / Consumable) |
| `item_type` | EItemType | 소분류 (Weapon / Chest / Potion) |
| `item_grade` | EItemGrade | 등급 (Common / Rare — 소비 아이템은 없음) |
| `item_base_stat` | int32 | 기본 성능 수치 (무기: STR 증가량, 갑옷: DEF 증가량) |
| `item_bonus_options` | TArray | Rare 등급에만 무작위로 부여되는 추가 옵션 목록 |

### UEquipmentComponent 데이터

| 필드명 | 타입 | 설명 |
| :--- | :--- | :--- |
| `equipped_weapon` | FEquipmentSlot | 현재 장착 중인 무기 슬롯 (비어있을 수 있음) |
| `equipped_chest` | FEquipmentSlot | 현재 장착 중인 갑옷 슬롯 (비어있을 수 있음) |
| `total_bonus_str` | int32 | 장착 장비로 증가한 총 공격력 합산 (실시간 반영) |
| `total_bonus_def` | int32 | 장착 장비로 증가한 총 방어력 합산 (실시간 반영) |
| `total_bonus_hp` | int32 | 장착 장비 추가 옵션으로 증가한 총 최대 HP 합산 |

### FEquipmentSlot 구조체

| 필드명 | 타입 | 설명 |
| :--- | :--- | :--- |
| `equipped_slot_type` | EEquipSlotType | 슬롯 타입 (Weapon / Chest) |
| `is_equipped` | bool | 장비 장착 여부 |
| `mounted_item` | FItemData | 장착된 아이템 데이터 (`is_equipped`가 false이면 null) |

### 장비 장착/해제 데이터 처리 흐름

**장착 (Equip)**
1. `mounted_item`에 아이템 데이터 할당 → `is_equipped = true`
2. `item_base_stat` 및 `item_bonus_options`를 파싱 → `total_bonus_str/def/hp`에 즉시 누적 → 플레이어 스탯 컴포넌트에 반영

**해제 (Unequip)**
1. 해당 슬롯 성능을 `total_bonus_str/def/hp`에서 차감 → 플레이어 스탯 갱신
2. `mounted_item = null`, `is_equipped = false`

## 5. MVP 개발 마일스톤

> **개발 우선순위 제안 (Priority)**
> 1. **1순위 (핵심 데이터 플로우):** 1단계와 2단계를 싱글플레이 모드에서 텍스트(`UE_LOG`)로 먼저 검증. 데이터가 인벤토리와 장비 창 사이를 정상적으로 오가는지 로그로 확인하는 것이 디버깅에 유리하다.
> 2. **2순위 (네트워크 동기화):** 데이터 흐름이 완벽해지면 3단계(Replication)를 적용하여 클라이언트-서버 간 패킷이 정상적으로 오가고 핵/변조가 불가능한 구조인지 체크.
> 3. **3순위 (인터페이스 확장):** 상호작용은 반드시 `UInterface`로 만들어야 추후 발판 메커니즘, 움직이는 장애물 등 환경 요소 확장 시 플레이어 코드를 수정하지 않고 대응할 수 있다.

| 단계 | 개발 프로세스 | 핵심 목표 | 예상 기간 |
| :--- | :--- | :--- | :--- |
| **1단계** | 데이터 기초 및 아이템 베이스 구축 | `FItemData` 구조체, DataTable, `AItemActorBase` | 1-2일차 |
| **2단계** | 핵심 컴포넌트 로직 구현 (Local) | 싱글플레이 기준 인벤토리/장비 로직 완성 | 3-7일차 |
| **3단계** | 서버구축 / 멀티플레이어 리플리케이션 적용 | 서버 검증 및 클라이언트 동기화 (RPC) | 8-10일차 |
| **4단계** | 상호작용 시스템 구축 | `IInteractionInterface` 기반 상호작용/취소 구현 | 11-14일차 |
| **5단계** | UI 및 최종 연동 | UMG 데이터 바인딩 및 폴리싱 | 15-17일차 |

### 1단계 — 데이터 기초 및 아이템 베이스 구축 (1-2일차)

- [ ] **아이템 데이터 구조 설계 (C++)**
  - 기획서의 Item 세부 명세를 바탕으로 `FItemData` 구조체 정의
  - 아이템 카테고리(`Equipment`, `Consumable`), 타입(`Weapon`, `Chest`, `Potion`), 등급(`Common`, `Rare`) `UENUM` 선언
  - 인벤토리 한 칸을 담당할 `FInventorySlot` 구조체(ItemData + Quantity) 정의
- [ ] **데이터 테이블(Data Table) 생성**
  - `FItemData`를 기반으로 에디터에서 사용할 DataTable 생성 및 샘플 데이터(무기, 갑옷, 물약) 입력
- [ ] **드롭 아이템 베이스 액터 생성 (`AItemActorBase`)**
  - 바닥에 떨어져 상호작용할 수 있는 아이템 액터 구현
  - 시각적 표현을 위한 `UStaticMeshComponent` 및 충돌 감지용 `USphereComponent` 추가

### 2단계 — 핵심 컴포넌트 로직 구현 (3-7일차)

- [ ] **인벤토리 컴포넌트 (`UInventoryComponent`)**
  - `TArray<FInventorySlot>` 구조로 30칸짜리 가방 공간 생성
  - 아이템 획득/추가 로직: 물약(소비형)은 기존 슬롯 수량(`quantity`) 증가, 장비형은 무조건 개별 빈 슬롯에 할당
  - 아이템 소실 로직: 수량이 0이 되면 슬롯 데이터를 비우는 기능 구현
- [ ] **장비 컴포넌트 (`UEquipmentComponent`)**
  - 무기 슬롯과 갑옷 슬롯을 나타낼 변수(또는 크기 2짜리 `TArray`/`TMap`) 정의
- [ ] **장착/해제 인터랙션 구현 (컴포넌트 간 통신)**
  - 장착(Equip): 인벤토리 슬롯 → 장비 슬롯 이동 및 기존 장비 스왑(인벤토리 빈 슬롯으로 복귀). 캐릭터 스탯(`STR`/`DEF`) 즉시 반영.
  - 해제(Unequip): 장비 슬롯 → 인벤토리 빈 슬롯 이동 및 캐릭터 스탯 차감. 인벤토리가 가득 찼을 때 해제 불가 예외 처리 필수.
- [ ] **소비 아이템(물약) 사용 로직**
  - 물약 사용 시 5초 동안 매초 전체 HP의 3~5%씩 서서히 회복되는 `FTimerHandle` 기반 타이머 구현

### 3단계 — 서버구축 / 멀티플레이어 리플리케이션 (8-10일차)

- [ ] **컴포넌트 및 인벤토리 배열 리플리케이션 설정**
  - 두 컴포넌트를 `bReplicates = true`로 설정
  - 인벤토리 슬롯 배열을 `UPROPERTY(ReplicatedUsing = OnRep_InventoryUpdated)` 형태로 지정하여 클라이언트 UI 갱신 이벤트와 연동 (더 정밀한 동기화가 필요하면 `FFastArraySerializer` 고려)
- [ ] **서버 검증 RPC (Server RPC) 구현**
  - 아이템 장착, 해제, 물약 사용 등 인벤토리 데이터가 변하는 모든 행동은 클라이언트가 서버에 요청(`Server_EquipItem`, `Server_UsePotion`)하도록 RPC 구현
  - 서버에서 인벤토리 공간 유효성 검증 후 데이터 변경 수행

### 4단계 — 상호작용 시스템 구축 (11-14일차)

- [ ] **상호작용 인터페이스 구현 (`IInteractionInterface`)**
  - C++ 인터페이스를 생성하여 `ExecuteInteraction(APlayerController* Interactor)` 순수 가상 함수 정의
  - `AItemActorBase`, NPC 액터, 구조물 액터가 이 인터페이스를 상속받아 각자 다른 효과(아이템 획득, 대화 창 오픈, 상자 열기 등)를 처리하도록 오버라이딩
- [ ] **플레이어 상호작용 레이캐스트/트레이스 로직**
  - 플레이어 캐릭터에서 Line Trace 또는 Sphere Overlap을 통해 1미터 이내의 상호작용 대상 감지
  - 대상이 감지되었을 때 F 키를 누르면 Server RPC를 통해 해당 대상의 `ExecuteInteraction` 호출
  - 아이템 마우스 좌클릭 획득: `GetHitResultUnderCursor`로 아이템 감지 → 획득 RPC 요청
- [ ] **상호작용 취소 조건 반영**
  - ESC 버튼: 현재 열린 UI 창 및 대화 상태 종료
  - 자동 취소(피격): `TakeDamage` 발생 시 진행 중인 상호작용 상태/NPC 대화 UI 강제 닫기

### 5단계 — UI 및 최종 연동 (15-17일차)

- [ ] **UI - C++ 데이터 바인딩**
  - 인벤토리 컴포넌트의 데이터 변경(`OnRep` 이벤트 등)이 발생하면 UMG 슬롯들이 새로고침되도록 이벤트 디스패처 연동
- [ ] **마우스 조작(우클릭/더블클릭) 기능 연동**
  - 슬롯 UI에서 더블클릭 또는 우클릭 감지 시, 플레이어 컨트롤러를 통해 서버에 장착/해제/사용 RPC 호출
